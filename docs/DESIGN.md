## What Makes This Cool

There is no public, reproducible, multi-policy leaderboard for LeRobot's sim envs today. Individual papers cite their own numbers on their own envs; comparisons across papers live in PDF tables nobody trusts. This artifact fills that gap with rigor — multi-seed, confidence intervals, reproducible eval — and pairs it with an interactive surface where a reviewer can browse rollouts in a single click. The wow moment is the *finding*: a non-obvious result that falls out of the comprehensive sweep, defended with statistics and visible in the video grid.

## Methodology (added per review)

**Episode definition.** An episode runs from `env.reset(seed=s_episode)` until either (a) `terminated=True`, (b) `truncated=True`, or (c) `step_count == max_steps`. Per-env `max_steps`:
- PushT: 300
- Aloha (each task): 400
- Libero (each task): 600

Success is the per-env standard reward threshold from `lerobot.envs.<env>.config.SUCCESS_REWARD` — not redefined here.

**Seeding contract.** Per cell `(policy, env, seed_idx ∈ {0..4})`:
- At cell start: `numpy.random.seed(seed_idx * 1000)`, `torch.manual_seed(seed_idx * 1000)`, `torch.cuda.manual_seed_all(seed_idx * 1000)`.
- Per episode `e ∈ {0..49}`: `env.reset(seed = seed_idx * 1000 + e)`. Policy stochasticity inherits the torch generator (NOT re-seeded per episode).
- **Resumability constraint:** because the torch generator advances across episodes within a cell, mid-cell resume is NOT bit-reproducible. `checkpointing.py` therefore resumes only at cell boundaries — if a process dies during a cell, that entire cell restarts from episode 0. This is documented in the methodology section of the public artifact.

**Per-env success thresholds.** Verified at Day 0 — read from `lerobot.envs.<env>.config.SUCCESS_REWARD` if it exists in v0.5.1, otherwise hardcoded in `envs.py` as: PushT 0.95 reward (coverage threshold), Aloha 1.0 (task-complete flag), Libero 1.0 (task-complete flag). Decided after a 5-min check in Day 0.

**Bootstrap CI.** For each cell, success rate = `successes / (seeds × episodes_per_seed)`. 95% CI via 10,000 bootstrap resamples over the flat list of `5 × n_episodes_per_seed` binary outcomes per cell (where `n_episodes_per_seed` is 50 for non-downscoped cells, may be lower per the auto-downscope rule below). Comparisons across cells report Δsuccess + 95% CI from the same bootstrap.

**Compute budget rule (auto-downscope).** Total per-cell wall-time budget is 3 hours. After Day 0 calibration produces per-policy `mean_ms_per_step`, episodes-per-seed for each cell = `min(50, floor(3 * 3600 / (5 * env_max_steps * mean_s_per_step)))`. If a policy's per-cell minimum (e.g., 5 episodes/seed × 5 seeds = 25 episodes total per cell) still exceeds 3 hours, the policy is dropped from v1, not silently truncated.

**Video render policy.** To keep dataset size sane and Spaces fetch latency tolerable: render all episodes per cell at 256px / 10fps / H.264 with a 2MB-per-MP4 size cap (re-encode and bitrate-clamp if exceeded). Total upper bound across the matrix is ~3-5GB of MP4 in the Hub dataset. Browse-rollouts UI uses `gr.Video(value=hub_direct_url)` — direct link from Hub, not proxied through Spaces — to keep CPU Space within memory limits.

**Sparse matrix policy.** A (policy, env) cell is N/A only when no public HF Hub checkpoint exists at lock-in time. Sparsity is reported in the methodology section, not silently skipped. Cross-policy comparisons are only made within shared envs.

## Approaches Considered

### Approach A — Static dashboard (minimal viable)
All results pre-computed locally on the RTX 4060. Pushed to HF Hub as a dataset. Spaces app is a Gradio reader rendering table + video grid from JSON. No interactive surface. Effort S (~3-5 days), risk Low. Rejected as too thin to differentiate.

### Approach B — Hybrid: static + browse-rollouts playground (RECOMMENDED)
A's full benchmark pipeline, plus Spaces hosts a "browse rollouts" tab: user picks (policy, env, seed, episode), the Space fetches the corresponding pre-recorded MP4 from the HF Hub dataset and renders it inline with metadata (success/fail, episode return, step count). Pre-computed video grid for the full leaderboard. **No live inference in v1** — the playground is a curated browser over the recorded sweep. Effort M (~7-10 days), risk Medium.

### Approach C — Continuous benchmark pipeline (lateral, deferred)
Reframed as automated MLOps: GitHub Actions runs the benchmark whenever a new policy tagged `lerobot-pusht` (etc.) appears on HF Hub. Spaces auto-updates. Becomes community-maintained infrastructure. Effort XL (~3-4 weeks), risk High. **Captured as v2 next-steps only — not started in this build.**

## Recommended Approach

**Approach B — Hybrid static benchmark + browse-rollouts playground.**

The "interactive playground" framing is downgraded from live inference (impossible on free CPU Spaces) to a curated rollout browser fed from the HF Hub dataset. This still beats Approach A's static table because the reviewer can click any cell and see the actual rollout, not just a grid of thumbnails — and it's honestly feasible without paid GPU.

## Open Questions

1. **Libero install on WSL.** Real risk. **Hard time-box: 4 hours.** If `pip install -e ".[libero]"` plus a single rollout doesn't work in 4 hours, drop Libero and proceed with PushT + Aloha. Resulting matrix is still 4 policies × 2 envs.
2. **VLA per-step inference latency.** This is the largest landmine — Pi0 and SmolVLA on RTX 4060 may run at ~0.5-2 s/step rather than the implicit ~30ms assumption. **Day 0 calibration spike** measures actual per-step latency for each policy on this hardware, then sets the achievable sweep size. If a VLA cell would take >3 hours, downscope episodes (50 → 25) for that policy only, documented in methodology.
3. **Pi0 VRAM at fp16.** Pi0 is ~3B params. fp16 = ~6GB, plus activations and KV cache. Likely fits but tight. If OOM at fp16, Pi0 is dropped from v1 (no quantization shipping). Decision after Day 0 spike.
4. **Compute budget for the sweep.** 4 (or 5) policies × 2-3 envs × 5 seeds × 50 episodes, accounting for VLA latency, sets the runtime. Hard ceiling: 24 hours of GPU time across 2 nights. If exceeded, episodes drop from 50 to 25 (premise 4 amended explicitly in the doc, not silently).
5. **Sweep resumability.** Per-cell checkpointing is mandatory. WSL2 + laptop sleep + flaky USB-C dock are real failure modes. `run_sweep.py` skips any (policy, env, seed) cell already present in `results.parquet`.
6. **No public benchmark exists today** — verify with 10 minutes of HF Hub + Google Scholar search before lockin. If something close exists, reposition (e.g., "extending X with seed-rigor and interactive surface").
